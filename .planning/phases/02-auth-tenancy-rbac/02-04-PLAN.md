Build user management endpoints (Admin CRUD across tenants, SM management within tenant), the approval chain system with configurable gates, and the audit log foundation that records all approval decisions.

Purpose: Admins need to manage users across the platform. SMs need to onboard Customer users in their tenants. The approval chain is the governance backbone — every shortlist submission and rate override must go through the right approver. The audit log captures these decisions for compliance.

Output: User management API, approval request workflow with decision tracking, audit log infrastructure, and tests for all flows.

    2. Create `backend/app/api/users.py` — FastAPI APIRouter with prefix="/users", tag="users":

       **Admin endpoints (use get_db superuser — cross-tenant):**

       a. `GET /users` — Admin only (Depends(require_roles(Role.ADMIN))):
          - List all users across all tenants
          - Optional query params: role (filter), tenant_id (filter), is_active (filter)
          - Uses `get_db` (superuser)
          - Return list of UserResponse

       b. `POST /users` — Admin only:
          - Accepts UserCreate body
          - Check email not already taken
          - Hash password with hash_password()
          - Create User with email_verified=True (admin-created users skip verification)
          - If tenant_id provided, verify tenant exists and is active
          - If role is "customer" or "candidate", tenant_id is required (they must belong to a tenant)
          - If role is "sm" or "recruiter" and tenant_id provided, also create UserTenantAssignment
          - Uses `get_db` (superuser)
          - Return UserResponse with 201

       c. `GET /users/{user_id}` — Admin only:
          - Get single user by ID
          - Uses `get_db` (superuser)
          - Return UserResponse or 404

       d. `PATCH /users/{user_id}` — Admin only:
          - Accepts UserUpdate body
          - Apply partial update
          - If role changed to customer/candidate, ensure tenant_id is set
          - Uses `get_db` (superuser)
          - Return UserResponse

       e. `POST /users/{user_id}/deactivate` — Admin only:
          - Set user.is_active = False
          - Revoke all refresh tokens for this user (set revoked_at=now)
          - Uses `get_db` (superuser)
          - Return UserResponse

       f. `POST /users/{user_id}/activate` — Admin only:
          - Set user.is_active = True
          - Return UserResponse

       **SM endpoints (tenant-scoped):**

       g. `POST /tenants/{tenant_id}/users` — SM only (for this tenant):
          - Accepts UserCreate body but only allows role="customer"
          - SM cannot create admin, sm, recruiter, or candidate users
          - Verify SM has assignment to this tenant via user_tenant_assignments
          - If SM is not assigned to this tenant, return 403
          - Create User with tenant_id from path, email_verified=True (SM-created)
          - Uses `get_db` (superuser for the write — we need to bypass RLS for creating users in another tenant than the SM's own)
          - Return UserResponse with 201

       h. `PATCH /tenants/{tenant_id}/users/{user_id}` — SM only (for this tenant):
          - SM can update display_name and is_active of Customer users in their tenant
          - Cannot change role
          - Verify SM is assigned to this tenant
          - Verify target user belongs to this tenant and has role="customer"
          - Return UserResponse
